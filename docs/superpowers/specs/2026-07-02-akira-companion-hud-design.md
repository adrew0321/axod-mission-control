# AKIRA Local Companion HUD — Design

**Date:** 2026-07-02
**Status:** Approved (design)
**Feature branch:** `feat/akira-companion-hud`
**Builds on:** [2026-06-29-akira-local-companion-design.md](2026-06-29-akira-local-companion-design.md)

## Summary

The AKIRA Local Companion currently runs **headless** on the laptop: a Node process
that dials outbound to the Mini over SSE, drives a persistent Playwright browser
(`navigate / read / type / click`), and returns a `blocked` result when a click is
hard-gated. Today that gate surfaces in the **Mini's web HUD** and approval
round-trips back through the Mini.

This project gives the Companion a **native local presence**: an always-on-top
Electron HUD on the laptop showing connection, presence, the pending-approval
queue, and security posture — collapsing to a draggable orb. Crucially, the
**hard-gate approval queue moves onto the laptop**: you approve/deny locally, and
the Mini no longer has to be reachable to resolve a gate.

## Goals

- A native, always-on-top HUD on the laptop that matches the AKIRA theme
  (`#7fdcff` cyan, `#ff5acf` magenta, glass-morphism, Inter).
- Full panel ⇄ minimized orb, draggable, launches alongside the Companion.
- **Local-first approvals:** the Companion holds the gate queue and surfaces it to
  the HUD over a localhost bridge; approve/deny/stop resolve locally.
- Real data only — no invented flavor fields.

## Non-Goals (explicit)

- **File transfer laptop↔Mini** — deferred to its own next slice (its own
  brainstorm → spec → plan). This project only lays the local-agent foundation
  that makes it smaller later.
- Launching native apps; multi-device / per-device tokens; password-manager
  integration; installers / autostart / code-signing (a later packaging slice).
- Any change to AKIRA's *capabilities* — the Companion's hands stay
  `navigate / read / type / click`.

## Architecture

Two local processes on the laptop, one localhost bridge between them:

```
                 Mini (bridge.axodcreative.com)
                        │  AKIRA's commands  (SSE, unchanged)
                        ▼
  ┌─────────────────────────────────────┐
  │  companion  (Node + Playwright)      │
  │  · connects out to Mini (as today)   │
  │  · classifyClick() → local gate queue│
  │  · NEW: localhost bridge server      │
  └──────────────┬──────────────────────┘
        localhost WebSocket (127.0.0.1 only)
   presence + gate queue  ▲   ▼  approve / deny / stop
  ┌──────────────┴──────────────────────┐
  │  Electron HUD app  (always-on-top)   │
  │  · full glass panel ⇄ minimized orb  │
  └──────────────────────────────────────┘
```

### Local bridge

- **Transport:** WebSocket server bound to `127.0.0.1` only (no inbound network
  exposure, no new outbound). Started by the Companion process.
- **Auth:** a short random token generated by the Companion at startup and written
  to a local file (e.g. `~/.akira-companion/bridge.token`, mode 600). The Electron
  HUD reads the same file and presents the token on connect. This prevents another
  local process from connecting to the bridge.
- **Direction of data:**
  - Companion → HUD: `presence` (connection state, uptime, operator, host, current
    task), `queue` (pending approvals), `security` (token/transport/profile/domains).
  - HUD → Companion: `approve{id}`, `deny{id}`, `stop`.
- **Degradation:** if the HUD is not running, the Companion still works headless
  and falls back to the existing Mini gate path (`/api/companion/approve`). If the
  Companion is down, the HUD shows a clear **disconnected** state (orb dimmed, no
  green dot) and reconnects automatically.

### Gate flow (changed)

Today `classifyClick()` already runs **inside the Companion** (`browser.ts`) and a
gated click returns `status: 'blocked'` — that result is posted straight to the
Mini and forgotten; approval must arrive later as a fresh `approved` click via
`/api/companion/approve`. The change is to **hold** the gated command locally
instead of dropping it, and resolve it on the laptop:

1. A command from the Mini reaches the Companion's one-at-a-time exec chain.
2. For a `click`, `browser.execute` still returns `blocked` when
   `classifyClick()` gates it. The exec chain treats that as "needs approval":
   it **enqueues the held command** in `gate-queue` and pushes it to the HUD,
   rather than immediately posting `blocked` to the Mini. Because the chain is
   one-at-a-time, it naturally waits on the operator's decision.
3. The operator taps **Approve** → the chain re-runs the same click with the
   approved flag and posts the resulting snapshot to the Mini. **Deny** → the held
   command is dropped and a declined `blocked` result is posted to the Mini so
   AKIRA knows.
4. **Timeout:** an un-actioned gate is treated as **declined** after a window
   (carried over from the Companion design spec's hard-gate-with-no-response rule).
5. **Stop:** aborts the current activity and clears the queue (the spec's kill
   switch), and closes the browser task in flight.

Existing Mini path (`/api/companion/approve` → registry → companion) remains as a
**fallback** for when the HUD isn't running; local is primary. When the HUD is
connected, the Companion holds gates locally; when it isn't, it falls back to
posting `blocked` to the Mini as today.

## HUD contents (mock → real data)

| Panel section | Real data source |
|---|---|
| Header `CONNECTED` + active-session timer | Companion↔Mini SSE link state + uptime since connect |
| **System Presence** | Operator name (config), `companion online`, laptop hostname, current browser task (idle / "reading example.com") |
| **Pending Approvals [n]** + Deny/Approve | The local gate queue: reason, target label + host, requested timestamp |
| **Connection & Security** | Token authenticated ✓, transport `outbound-only`, browser profile `persistent · local`, N sensitive domains guarded |
| **Stop** kill-switch (footer) | Sends `stop` over the bridge |

**Minimized orb:** always-on-top badge; green dot when connected, dimmed when
disconnected, pulses/turns magenta when ≥1 approval is waiting. Click to expand;
drag to reposition.

**Dropped from the original mock as invented flavor:** profile photo/rank
(KANEDA/SHOTARO, CAPTAIN/PILOT), literal "AES-256 / MFA ENABLED" strings, and
per-device multi-session. Replaced by the real signals above.

## Components / units

Each unit has one purpose and a testable boundary.

- **`companion/src/gate-queue.ts`** — pure queue: enqueue a held command, list
  pending, resolve by id (approve/deny), expire on timeout. No I/O. Unit-tested.
- **`companion/src/bridge.ts`** — the localhost WS server: auth via token file,
  serialize `presence`/`queue`/`security` snapshots to connected HUD clients,
  accept `approve`/`deny`/`stop`. Thin; delegates decisions to `gate-queue` and
  callbacks into `index.ts`.
- **`companion/src/bridge-protocol.ts`** — shared message types for the bridge
  (mirrors the existing `protocol.ts` duplication convention).
- **`companion/src/index.ts`** — wire `classifyClick` + `gate-queue` into the exec
  chain; start the bridge; expose `stop` and current-task/presence to it.
- **`companion-hud/`** — new Electron app:
  - `main` — always-on-top frameless window, size/position persistence, tray/quit,
    drag, minimize↔restore, reads the bridge token, connects to the localhost WS.
  - `renderer` — the HUD UI (full panel + orb) using the AKIRA theme tokens.
    Reuses the visual language from `src/components/akira` (colors, glass, orb feel)
    but is its own standalone renderer (no Next.js).
  - Excluded from the root `tsconfig.json` / `next build`, exactly like
    `companion/` is (laptop-only, never seen by the Mini build).

## Testing

- **Automated (in the `tsx --test` suite):** `gate-queue` (enqueue / list /
  approve / deny / timeout-expiry) and `bridge` message
  serialization/auth-rejection logic — **pure logic only**, no Electron, no live WS
  sockets, consistent with the project's testing convention.
- **Manual / integration (not automated):** Electron window behavior, live
  localhost WS handshake, and the end-to-end gate: start Companion + HUD → ask
  AKIRA to attempt a buy → gate appears in the local HUD → Approve runs it, Deny
  drops it and reports back to the Mini → Stop aborts.

## Error handling

- HUD can't reach the bridge → auto-reconnect with backoff; show disconnected
  state; Companion unaffected (headless + Mini fallback).
- Bridge token mismatch → connection refused; HUD surfaces a clear "auth failed"
  state rather than silently hanging.
- Companion crash/quit → HUD shows disconnected; on Companion restart the token
  file is rewritten and the HUD reconnects.
- Duplicate/again-gated approvals resolved idempotently by queue id.

## Project conventions honored

- Branch off `dev` (`feat/akira-companion-hud`); merge to `dev` when green; **do
  not** push to `main` / tag / deploy (release is a separate human-gated step).
- `pnpm`, `ignore-scripts=true` left as-is; extensionless TS imports; tests via
  `tsx --test`.
- `companion/` and `companion-hud/` stay excluded from the root build; Playwright
  and Electron never reach the Mini.
- Work in the main checkout on the feature branch (no linked worktree — Turbopack
  breaks on the junctioned `node_modules`).

## Deploy note (for the human, not the build)

The Mini needs no change for this feature (approvals go local; no DB migration,
no new Mini env var). The HUD is a laptop-side app run alongside the Companion.
Packaging/installer/autostart is a later slice.
